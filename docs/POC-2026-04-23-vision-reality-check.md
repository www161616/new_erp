# POC 2026-04-23 — Claude Vision 對 new_erp 的實戰評估

## TL;DR

1. **AI 投資方向大轉彎**：從「拍東西讓 AI 看」 → **「解析已存在的結構化資料」**（ROI 差 10 倍以上）
2. **3 個高 ROI spike**：團購記事本貼文解析 / 1688 商品頁解析 / LINE 記事本按讚名單解析
3. **2 個反模式確認**：商品實物照辨識（70~80% 無標籤）、手寫發票 OCR（錯誤率高）
4. **2 個業務事實揭露**：下單都在 LINE 記事本（不在聊天）、每天開團 21-30 個、70~80% 商品無標籤

---

## 測試背景

**目的**：驗證訂單 PRD §14「Claude Haiku vision 解析準確率 spike」的可行性與準確率上限。

**模型**：Claude Opus（不是 Haiku）— 所以結果是 **上限值**。Opus 做不到 → Haiku 鐵定不行；Opus 做得到 → Haiku 大概率也做得到（部署前需再跑一次實機驗證）。

**方法**：使用者提供真實截圖 / 照片 → Read tool 直接讀進 Claude 視覺 context → 結構化輸出 → 使用者評準確率。

## 7 張測試素材與結果

### 1️⃣ LINE 記事本貼文（顧客端、舊樣本）
**檔案**：`螢幕擷取畫面 2026-04-23 144417.png`
**結果**：
- 辨識到 27 個讚的部分用戶列表
- 暱稱格式不統一（如 `Janett8252228456中和環峰`），但店名 hint 可拆出（中和 / 環峰）
- **⚠️ 揭露業務問題**：LINE 記事本按讚式下單**沒有數量資訊** → 預設 qty=1 + 留言補充
**準確率**：⭐⭐⭐（按讚名單 OK、但業務資料本身不足）

### 2️⃣ 電子發票（德麥食品 YY-05871504）
**檔案**：`92736.jpg`
**結果**：發票號、期別、開立時間、賣方/買方統編、3 筆商品明細、稅額、總計、付款方式 — **全部正確抽出**。數字一致性檢查：4725+4771+2505=12001、12001×5%=600、總計 12601、件數 26 = 11+10+5 ✅
**準確率**：⭐⭐⭐⭐⭐
**⚠️ 反模式警告**：電子發票**有 QR code**、應走**財政部 API 拉取**而非 vision；vision 應留給紙本三聯式 / 收據。

### 3️⃣ 手寫發票（三聯式 XV 46333226）
**檔案**：`92740.jpg`
**結果**：發票號、期別、日期、營業人統編可辨；但品名、手寫數字、中文大寫金額、印章模糊位置 = 低信心；須人工 review。
**準確率**：⭐⭐ 中低
**結論**：可行但不能全自動；需「AI 草稿 + 人工確認」UI 流程。符合記憶中「分散型供應商、vision ROI 低」反模式預期。

### 4️⃣ 商品實物照（MIZU 牙膏 × 4 張）
**檔案**：`新增資料夾 (2)\S__4368003{6..9}_0.jpg`
**結果**：
- 品牌 MIZU / 商品名（中日英三語）/ 規格 110g / 條碼 4920754683136（EAN-13）/ 批號 G8763 / MFG 2026-01-10 / EXP 2029-01-09 / 製造地 日本 / 輸入商 安辰有限公司 / 電話 / 成分 / 使用方法 — **全部正確**
**準確率**：⭐⭐⭐⭐⭐
**⚠️ 但這是萬分之一的情境**：
- 使用者後續揭露：每天開團 21-30 樣、商品組成 ~70-80% **無標籤**（生鮮 / 陸貨）
- 這張 MIZU（有詳細多語標籤）= 日韓美保健食品類的代表、僅佔 ~20%
- **大部分商品照無法用 vision 抽出結構化資料**
**結論**：商品實物照 AI 識別只對 ~20% 有標籤品類有效，**不適合作為主要建檔入口**。

### 6️⃣ 1688 下單確認截圖

## 業務面重大發現（POC 過程中揭露）

### 發現 1：LINE 社群下單都在**記事本**，不在聊天
- 聊天訊息裡**不會有人喊 `+N`**
- 顧客下單 = 在記事本貼文**按讚**（預設 qty=1）、需要 >1 份才留言寫數量
- **影響**：訂單 PRD §7.3 的「`+N` 留言解析」假設**不成立**
- **記憶**：`project_new_erp_line_order_reality.md`

### 發現 2：**70-80% 商品無標籤**，每天 21-30 團
- 生鮮蔬果（完全無包裝）~30%
- 陸貨生活用品（大多無標籤）~40%
- 日韓美保健食品（有詳細標籤）~20%
- 其他（糕餅 / 零食）~10%
- **影響**：商品 PRD Q13「邊用邊建」是對的方向；但 AI 照片識別**只對 ~20%** 品類有用
- **記憶**：`project_new_erp_product_reality.md`

### 發現 3：AI 投資方向應**轉向「解析已結構化資料」**
- ❌ 拍實體商品讓 AI 看：70-80% 沒字可讀
- ❌ 拍手寫紙本發票：分散格式、錯誤率高
- ✅ 解析**你自己寫的團購貼文**：格式穩定、AI 準確率高
- ✅ 解析**1688 商品頁**（已是結構化 HTML）：穩定欄位
- ✅ 解析**LINE 記事本按讚**（+ 留言補數量）：至少比純人工省

## 建議：3 個真實 POC Spike

| # | 名稱 | 輸入 | 輸出 | 首個里程碑 |
|---|---|---|---|---|
| **Spike-1** | 團購記事本貼文解析 | LINE 記事本貼文（使用者自寫）| `{campaign_name, items[], price, variants[], close_date, spec, description}` | 10 篇真實樣本、AI 抽取 ≥95% 欄位正確 |
| **Spike-2** | 1688 / 拼多多商品頁解析 | 商品頁截圖（原解析度）| `{products, skus, sku_suppliers}` 草稿 | 10 個不同品類商品、關鍵欄位正確率 ≥90% |
| **Spike-3** | LINE 記事本按讚名單解析 | 按讚列表截圖 | `[{暱稱, 取貨店 hint, 按讚時間}]` + qty=1 預設 | 5 張實際記事本、暱稱辨識 ≥95% + 店名 hint 正確率 ≥80% |

各 Spike **執行建議**：
- 用真正的 Haiku 跑（不是 Opus）、記錄 p50/p95 latency + 每次成本
- 準備 **golden dataset**（20~50 張已人工標註的樣本）評準確率
- 失敗案例歸類（pattern），判斷是 prompt 不夠還是模型能力不足

---

## 不建議投資的方向（反模式）

- ❌ **拍實體商品自動建檔**：70-80% 無標籤、ROI 極低
- ❌ **手寫紙本發票 OCR 入帳**：格式雜、錯誤率高、量又分散
- ❌ **電子發票 vision OCR**：走財政部 API 更乾淨
- ❌ **LINE 聊天訊息 `+N` 解析**：顧客根本不在聊天下單
