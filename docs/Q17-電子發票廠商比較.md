## 核心設計（per-store 發票模式，2026-04-23 新增）

依 [[decisions/2026-04-23-系統立場-混合型]] C 混合型、加盟店自主決定發票模式：

| 模式 | 意思 | 適用 |
|---|---|---|
| `enabled` | ezPay 子商店電子發票 | 營業額 > 20 萬、或加盟主主動啟用 |
| `manual` | 紙本 / 發票機 | 規模中等、不想走電子 |
| `none` | 合法免開（預設）| < 20 萬 |

**on-demand 開通**：加盟主切到 `enabled` → 系統呼叫 ezPay 主帳號 API 建立子商店（不預先簽 100 個）。
**法遵監控**：月營業額 ≥ 80% × 20 萬 → 觸發 `invoice_threshold_warning` 通知（見 [[PRD-通知模組]] §6.1）。

詳細規格：[[PRD-訂單取貨模組]] §7.12。

## 100 家加盟店試算（更新）

**新假設**（2026-04-23 修正）：
- 不是「100 家全部立即上線電子發票」
- 是「**on-demand** 模式：每家依自己營業額決定是否切 `enabled`」
- 保守假設前 2 年約 30~50 家達到 20 萬門檻、其餘 `none`

### 年度 TCO 估算（ezPay 擴容）

| 情境 | 子商店數 | 方案假設 | 年度估算（未稅）|
|---|---|---|---|
| **v1 pilot**（10 家先行）| 10 | 基本型 × 10 | $60,000 |
| **pilot 後 6 個月**（30 家）| 30 | 基本型 × 30 | $180,000 |
| **穩定期**（50 家達門檻）| 50 | 基本 × 40 + 菁英 × 10 | $360,000 |
| **全量**（100 家全部走 enabled）| 100 | 基本 × 70 + 菁英 × 30 | $780,000 |

**對比原先 v0.1 假設（100 家全部立即上線 $1,200,000/年）**：真實成本會**明顯更低**。

**待 ezPay 業務確認後更新**：老客戶擴容折扣、子商店計費時機、主帳號量大合併議價。

## 每家廠商要點

### 🥇 ezPay 簡單付（首推 — 擴容）
- **關係優勢**：3 年既有客戶、員工熟悉後台、UI 親民對婆婆媽媽友善
- **轉換成本 = 0**：不用合約重簽、不用重新訓練、不用重新串接 API
- **現有方案**：總部基本型 $4,800/年（月 100 張 B2B 用）— 運作穩定
- **擴容關鍵未知**：
  - 主帳號 + 子商店架構（最關鍵、決定 on-demand 擴展可行性）
  - 老客戶擴容折扣（談判空間）
  - 子商店計費時機（簽約就扣 vs 第一張才扣，on-demand 對後者才有意義）
- **談判節奏**：**你就是你們 3 年客戶**，講話擺姿態；條件太差再找綠界比價

### 🥈 綠界 ECPay（備案 — 比價）
- **最大賣點**：2026/6/30 前新戶首年 0 元、20,000 張；設定費 $3,600 僅收一次
- **續年費最便宜**：$6,000 / 20,000 張
- **SDK / 文件最完整**、社群範例多、工程整合成本低（若真的要換廠商）
- **為什麼降級為備案**：
  - 轉換成本（重簽、重訓、重串）＞ 新戶優惠金額
  - 總部已習慣 ezPay 後台、員工訓練不用再來一輪
- **什麼時候啟動**：ezPay 擴容條件**明顯差**（沒主帳號子商店、或擴容報價高於綠界新戶優惠 1.5 倍以上）才回頭談

### ❌ 藍新 / 永豐 / 關貿（跳過）
同 v0.1 理由（綁金流 / KYC 太高 / 殺雞用牛刀）。

## 對「100 家加盟店」場景的建議（v0.2）

1. **先打 ezPay 現有業務窗口**（見 `d:\LT-ERP 專案\Q17-ezPay電話備忘.md`）— 3 年老客戶、不是 cold call
2. **關鍵 3 題確認**：主帳號+子商店架構 / on-demand 開通 SLA / 老客戶擴容折扣
3. **per-store 發票模式**（v1 核心設計）：加盟主自主決定模式、`none` 也有法遵門檻監控
4. **系統預備 ezPay 子商店 API 對接**：即使 v1 pilot 只開 10 家、架構一開始就建好
5. **若 ezPay 條件不理想 → 回頭打綠界**（02-2655-1775，搶 2026/6/30 新戶優惠作為 hard negotiation leverage）

## 還不確定、需會計師/業務確認的點

對應 [[Q17-會計師諮詢清單]] §5（電子發票 API 串接）、§9（成本預估）：

### ezPay 業務（優先）
- [ ] **主帳號 + 子商店架構是否存在**（最關鍵、決定擴展可行性）
- [ ] **老客戶擴容折扣**（我方 3 年關係籌碼）
- [ ] **子商店 on-demand 新增**（不一次簽 100 個）
- [ ] **子商店計費時機**（簽約 vs 第一張）
- [ ] **張數上限層級**（各子商店各算 vs 主帳號彙總）
- [ ] **現有基本型升級成主帳號**是否可行
- [ ] **API 測試環境 / 折讓作廢費用 / SLA 賠償條款**

### 會計師
- [ ] 加盟店獨立稅籍分工（100 家各自稅籍 vs 總部代開的法律界線）
- [ ] 員工團購（§7.10）發票：月結彙總 vs 每次開立
- [ ] 退貨折讓單跨期處理
- [ ] 過渡期（v1 部分店 `none`）法律風險
- [ ] 7 年稅法保存責任：廠商 vs 總部 ERP

### 綠界（只有 ezPay 破局才問）
- [ ] 多統編計費結構（100 個統編 = 100 份合約？）
- [ ] 「新戶」認定：總部用過綠界金流者是否算
- [ ] 量大折扣：120 萬張/年的企業整批價
